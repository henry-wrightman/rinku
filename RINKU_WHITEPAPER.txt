Abstract

Rinku is a DAG-based distributed ledger optimized for private, ephemeral
communication channels that remain cryptographically verifiable. Unlike
traditional blockchains that treat privacy as an afterthought, Rinku's
architecture is built so that each layer — from self-contained proofs to
sub-second finality to encrypted messaging — reinforces a single thesis:

    Participants should be able to prove what they need to prove
    without revealing anything they didn't choose to share.

Put differently: Rinku is a ledger that produces portable,
offline-verifiable receipts. The DAG and checkpoints are the
production line; the receipt is the product. Everything else —
fast finality, encrypted messaging, mailbox addresses, closed-network
boundaries, stake-weighted trust — exists to make those receipts
private, fast, and trustworthy.

This paper describes Rinku's privacy architecture across five layers:

    I.   Foundation  — DAG + checkpoints + self-contained proofs
    II.  Speed       — Mysticeti-FPC fast-path finality
    III. Privacy     — Encrypted channels, mailbox addresses, closed networks
    IV.  Trust       — Weight attestation (accountability without identity)
    V.   Economics   — Tokenomics that sustain the validator network

================================================================================
            I. FOUNDATION: VERIFIABLE STATE WITH MINIMAL DISCLOSURE
================================================================================

The foundation layer establishes a ledger where verification never
requires more state than the verifier chooses to request. Every proof
is self-contained. Every verification is offline-capable.

The network exists to produce portable receipts — not to be queried.


1.1 URL-NATIVE STATE
--------------------------------------------------------------------------------

Rinku's fundamental innovation is embedding cryptographic state proofs within
URLs themselves. Each proof is self-contained and verifiable without external
node access.

This matters for privacy: a proof-carrying URL reveals exactly one fact
about the ledger (e.g., "this account had this balance at this checkpoint")
without exposing any other state. The verifier learns only what the prover
chose to share.

    Traditional Blockchain          vs          Rinku URL-Native
    ----------------------                      ----------------
    
    +------------+                             +------------------------+
    | Full Node  | <-- query                   |  /sp/{proof_payload}   |
    +------------+                             +------------------------+
         |                                              |
         v                                              v
    [ Verify via                               [ Proof contains all data
      node's state ]                             for offline verification ]
    
    Verifier sees:                             Verifier sees:
    Whatever the node                          ONLY the specific claim
    chooses to expose                          embedded in the proof


URL Format (path-based):
    /{profile}/{base64url(deflate(payload))}

Profiles:
    - /tx/{payload}    Transaction data
    - /ap/{payload}    Account proof
    - /sp/{payload}    Self-contained proof (full offline verification)
    - /wp/{payload}    Weight proof
    - /sc/{payload}    Smart contract deploy/call

Note: URLs are path-based for web compatibility. Presentation layers may
use `rinku://` scheme for deep-linking in native apps.


1.2 DAG-BASED TRANSACTION ORDERING
--------------------------------------------------------------------------------

Transactions form a Directed Acyclic Graph (DAG) rather than a linear chain.
Each transaction references 1-3 "tip" transactions, creating a partial order.

                        +-------+
                        | Tx A  |
                        +---+---+
                           / \
                          /   \
                   +-----+     +-----+
                   | Tx B|     | Tx C|
                   +--+--+     +--+--+
                      \           /
                       \         /
                        +---+---+
                        | Tx D  |  <-- references B and C
                        +-------+

Benefits:
  - Parallel transaction processing
  - Higher throughput than linear chains
  - Natural conflict detection via ancestry

The DAG structure is also essential for private communication: messages
(encrypted data-only transactions) can be submitted concurrently without
blocking on a linear ordering bottleneck, enabling real-time chat and
threaded conversations at protocol speed.


1.3 CHECKPOINT ARCHITECTURE
--------------------------------------------------------------------------------

Checkpoints are periodic snapshots that finalize batches of transactions.
They are created every ~10 seconds by an elected leader and signed by
validators via BLS aggregate signatures.

    struct Checkpoint {
        height:               u64         // Sequential checkpoint number
        hash:                 String      // SHA256 of checkpoint contents
        previous_hash:        Option<String> // Link to prior checkpoint (None for genesis)
        tx_merkle_root:       String      // Root of finalized tx Merkle tree
        state_root:           String      // Root of account state trie
        receipt_root:         String      // Root of execution receipts
        tip_count:            u32         // DAG tips at snapshot
        timestamp:            u64         // Unix ms
        validator_signatures: [ValidatorSignature] // Individual validator sigs
        aggregated_signature: Option<String>       // BLS12-381 aggregate sig (hex)
        signer_bitmap:        Option<[u8]>         // Which validators signed
        finalized_tx_hashes:  [String]    // Transactions in this checkpoint
        weight_trie_root:     String      // Root of weight attestations
    }
    
    struct ValidatorSignature {
        validator:       String      // Validator address
        signature:       String      // Signature (hex)
        weight:          f64         // Validator's stake weight
        bls_public_key:  Option<String> // BLS pubkey (hex, optional)
    }

Checkpoints serve as trust anchors for proofs. A self-contained proof
references a specific checkpoint, and the checkpoint's BLS aggregate
signature proves that a quorum of validators attested to that state.


1.4 STATE ROOTS & MERKLE PROOFS
--------------------------------------------------------------------------------

State is organized in a Sparse Merkle Trie (256 levels) enabling efficient
proofs for any account.

                          STATE ROOT
                              |
                    +---------+---------+
                    |                   |
               [subtree 0...]      [subtree 1...]
                    |                   |
              +-----+-----+       +-----+-----+
              |           |       |           |
           [...]       [...]   [...]       [...]
                                              |
                                    +---------+---------+
                                    |                   |
                                [Account A]        [Account B]

Leaf Hash Format (1 RKU = 100,000,000 micro-RKU):
    SHA256("account:{address}:{balance_micro}:{nonce}:{staked_micro}")
    
    Example: "account:abc123...def:1000000000:5:500000000"
             (10 RKU balance, nonce 5, 5 RKU staked)

Internal Node Hash:
    SHA256("node:{left_hash}:{right_hash}")

The Sparse Merkle Trie is privacy-preserving by construction: proving
one leaf reveals nothing about sibling leaves. A verifier who receives
an account proof learns only that one account's state — not who else
is in the trie, how many accounts exist, or what their balances are.


1.5 CHECKPOINT CREATION PROCESS
--------------------------------------------------------------------------------

    +-----------------------------------------------------------------+
    |  CHECKPOINT CREATION TIMELINE (~10 second intervals)            |
    +-----------------------------------------------------------------+
    
    t=0s     Leader elected (deterministic from prev checkpoint hash)
             |
    t=0-9s   Transactions accumulate, fast-path confirmations
             |
    t=10s    Leader creates checkpoint proposal:
             |   1. Collect finalized transactions (fast-path confirmed)
             |   2. Compute tx_merkle_root
             |   3. Execute transactions, update account states
             |   4. Compute state_root from account trie
             |   5. Finalize pending weight votes -> weight_trie_root
             |   6. Generate precomputed proofs for affected accounts
             |   7. Sign with BLS key
             |
    t=10s+   Broadcast CheckpointVoteRequest to peers
             |
    t=10-11s Validators verify & sign, return votes
             |
    t=11s    Quorum reached (>2/3 stake), checkpoint finalized
             |
    t=11s+   Broadcast CheckpointAnnouncement with proofs


================================================================================
              II. SPEED: REAL-TIME PRIVATE COMMUNICATION
================================================================================

Privacy without speed is email. Privacy with speed is conversation.
Mysticeti-FPC fast-path finality gives Rinku sub-second transaction
confirmation, making encrypted real-time communication practical as a
native ledger application rather than a layer bolted on top.


2.1 MYSTICETI-FPC FAST-PATH MECHANISM
--------------------------------------------------------------------------------

Rinku implements Mysticeti-style Fast-Path Confirmation (FPC) for sub-second
transaction finality. This bypasses the ~10s checkpoint cycle for immediate
user feedback.

ALL transaction types are eligible for fast-path — there is no "slow path"
subset. Validators only ACK transactions that pass full validation
(correct balance, nonce, signature, gas).

Operational Envelope:

    +-------------------------------------------------------------------+
    |              FAST-PATH OPERATIONAL ENVELOPE                        |
    +-------------------------------------------------------------------+
    | Parameter                | Value / Behavior                        |
    +--------------------------+-----------------------------------------+
    | Eligible transactions    | 100% (all types: Transfer, DataOnly,    |
    |                          | Stake, Unstake, ClaimRewards, Contract) |
    +--------------------------+-----------------------------------------+
    | Happy-path latency       | 100-500ms (no contention, quorum        |
    |                          | reachable, network healthy)             |
    +--------------------------+-----------------------------------------+
    | Contention behavior      | Conflicting transactions (same nonce    |
    |                          | from same sender) receive ACKs only for |
    |                          | the first-seen version. The conflict is |
    |                          | resolved by DAG cumulative weight at    |
    |                          | checkpoint time.                        |
    +--------------------------+-----------------------------------------+
    | Adversarial scheduling   | If <2/3 stake is online or responsive,  |
    |                          | fast-path cannot reach quorum. Tx falls |
    |                          | back to checkpoint-only finality (~10s).|
    +--------------------------+-----------------------------------------+
    | Fallback latency         | ~10-11s (next checkpoint cycle). No tx  |
    |                          | is lost — fast-path is an acceleration, |
    |                          | not a requirement for inclusion.        |
    +--------------------------+-----------------------------------------+
    | Backpressure             | When DAG tips exceed threshold, API     |
    |                          | returns HTTP 503. Fast-path ACKs are    |
    |                          | paused until tip count normalizes.      |
    +--------------------------+-----------------------------------------+

    +-------------------------------------------------------------------+
    |              FAST-PATH vs CHECKPOINT FINALITY                      |
    +-------------------------------------------------------------------+
    
    User submits tx
          |
          v
    +-----+-----+                    
    | Validator | -----> Broadcasts tx to all validators
    | (Entry)   |        with signed ACK
    +-----+-----+
          |
          +-----> Other validators receive tx
          |       Each validates & broadcasts ACK
          |
          v
    +-----+-----+
    | Quorum    |  <-- When >2/3 stake ACKs received
    | Reached!  |
    +-----+-----+
          |
          v
    FAST-PATH CONFIRMED (100-500ms)
          |
          | (tx still enters next checkpoint for
          |  state root inclusion & full finality)
          v
    CHECKPOINT FINALIZED (~10s)


2.2 QUORUM REQUIREMENTS
--------------------------------------------------------------------------------

    Quorum Threshold: 66.7% (2/3) of total active validator stake
    
    Example with 3 validators (1000 RKU each = 3000 total):
    
    +------------+--------+-------------+
    | Validator  | Stake  | ACK Status  |
    +------------+--------+-------------+
    | Val-A      | 1000   | ACK         |
    | Val-B      | 1000   | ACK         |
    | Val-C      | 1000   | (pending)   |
    +------------+--------+-------------+
    
    ACKed stake:  2000 RKU
    Threshold:    3000 * 0.667 = 2000 RKU
    Result:       QUORUM REACHED! (2000 >= 2000)

Fast-path also supports regular stakers, not just validator nodes.


2.3 FINALITY TIMELINE
--------------------------------------------------------------------------------

    +-------+------------+--------------------------------------------+
    | Time  | Event      | Description                                |
    +-------+------------+--------------------------------------------+
    | 0ms   | TX Submit  | User signs & submits transaction           |
    | 10ms  | Broadcast  | Entry validator broadcasts with ACK        |
    | 50ms  | ACKs       | Other validators validate & ACK            |
    | 100ms | Quorum     | 2/3 stake reached, FAST-PATH CONFIRMED     |
    | ~10s  | Checkpoint | Transaction included in checkpoint         |
    | ~11s  | Final      | Checkpoint signed by quorum, FULLY FINAL   |
    +-------+------------+--------------------------------------------+

Fast-path provides UX-level confirmation; checkpoint provides
cryptographic proof suitable for self-contained proofs.


2.4 WHY SPEED ENABLES PRIVACY
--------------------------------------------------------------------------------

Fast-path finality is not just a UX improvement — it is a privacy
prerequisite. Here's why:

    +-------------------------------------------------------------------+
    |              SPEED AS A PRIVACY ENABLER                            |
    +-------------------------------------------------------------------+
    
    Without fast-path (~10s finality):
    - Chat messages wait 10+ seconds for confirmation
    - Users resort to off-chain relays (breaking privacy model)
    - Threaded conversations become impractical on-chain
    - Ephemeral communication channels can't compete with centralized apps
    
    With fast-path (100-500ms finality):
    - Encrypted messages confirm in real-time
    - No incentive to bypass the ledger (and its privacy guarantees)
    - Threads and DMs are native ledger applications
    - Ephemeral channels feel like instant messaging
    
    +-------------------------------------------------------------------+

The critical insight: if finality is slow, users will route private
messages through centralized relays to get speed — and those relays
become surveillance points. Fast finality keeps private communication
on the ledger, where the privacy guarantees actually hold.

Validator gossip solves the relay problem: messages propagate through
the validator network via GossipSub, meaning no single relay sees all
traffic. Combined with encrypted content, validators transport messages
they cannot read.

                   III. PRIVACY: THE COMMUNICATION LAYER

3.1 THE PRIVACY THESIS
--------------------------------------------------------------------------------

Rinku's privacy model rests on a principle:

    +-------------------------------------------------------------------+
    |                                                                    |
    |  The ledger proves facts. It does not expose them.                 |
    |                                                                    |
    |  A participant can prove they sent a payment, received a           |
    |  message, or hold a balance — without revealing their identity,    |
    |  their other transactions, or the content of their communications. |
    |                                                                    |
    +-------------------------------------------------------------------+

This is achieved through four reinforcing mechanisms:

    1. SELECTIVE DISCLOSURE (Self-Contained Proofs)
       Prove exactly one fact without exposing any other state.
       
    2. CONTENT PRIVACY (Double Ratchet Encryption)
       Message content is encrypted end-to-end. Validators transport
       ciphertext they cannot decrypt. Forward secrecy ensures past
       messages stay private even if keys are later compromised.
       
    3. IDENTITY PRIVACY (Mailbox Addresses)
       Ephemeral receive addresses decouple identity from reachability.
       A sender addresses a mailbox, not a person.
       
    4. NETWORK PRIVACY (Closed-Network Boundaries)
       Closed networks limit the set of participants, reducing metadata
       leakage and making traffic analysis harder.

Together, these create communication channels that are:
  - Partially stateless (clients hold only their own proofs)
  - Ephemeral (mailbox addresses and key ratcheting limit linkability)
  - Localized (closed-network boundaries contain metadata)
  - Verifiable (every claim is backed by a cryptographic proof)


3.2 SELF-CONTAINED PROOFS AS SELECTIVE DISCLOSURE
--------------------------------------------------------------------------------

A self-contained proof includes ALL data required for verification of a
single claim — and nothing more. This is selective disclosure at the
protocol level.

  +-------------------------------------------------------------------+
  |                     SELF-CONTAINED PROOF                          |
  +-------------------------------------------------------------------+
  |  Transaction Data          Checkpoint Anchor                      |
  |  - hash, from, to          - height, hash                         |
  |  - amount, nonce           - state_root                           |
  |  - signature               - tx_merkle_root                       |
  +-------------------------------------------------------------------+
  |  Merkle Inclusion Proof    Validator Set Commitment               |
  |  - proof path              - MerkleSumTree root                   |
  |  - sibling hashes          - signer leaves + weights              |
  |  - leaf index              - BLS aggregated signature             |
  +-------------------------------------------------------------------+

No external data required. Verifiable offline.

    PRIVACY PROPERTY: When you verify a self-contained proof, you learn
    exactly one fact (e.g., "transaction X was finalized at checkpoint Y").
    You do NOT learn:
      - What other transactions exist in that checkpoint
      - What other accounts exist in the state trie
      - Who else is communicating on the network
      - The content of any encrypted messages
    
    The Merkle proof path contains sibling hashes — but these are opaque
    digests that reveal nothing about the data they commit to.


3.2.1 Account State Proofs
--------------------------------------------------------------------------------

An AccountStateProof allows verification of an account's balance, nonce, and
staked amount at a specific checkpoint without querying the network.

    struct AccountStateProof {
        version:              u32         // Proof format version (v3)
        address:              String      // Account identifier (40 hex chars)
        balance_micro:        u64         // Balance (1 RKU = 100,000,000 micro)
        nonce:                u64         // Transaction count
        staked_micro:         u64         // Staked (1 RKU = 100,000,000 micro)
        pending_rewards_micro:u64         // Unclaimed rewards
        staked_at:            u64         // Stake timestamp (ms)
        last_reward_at:       u64         // Last reward claim (ms)
        claimed_total_micro:  u64         // Lifetime claimed rewards
        
        // Checkpoint anchor
        checkpoint_height:    u64
        checkpoint_hash:      String
        state_root:           String
        
        // Merkle inclusion proof
        merkle_proof:         [String]    // Sibling hashes
        leaf_index:           u64         // Position in tree
    }

Verification Steps:
    1. Reconstruct leaf: SHA256("account:{addr}:{bal}:{nonce}:{staked}")
    2. Walk merkle_proof, hashing with siblings
    3. Compare computed root against checkpoint's state_root
    4. Verify checkpoint BLS signature (optional, for full trust)


3.2.2 Transaction Inclusion Proofs
--------------------------------------------------------------------------------

Prove a transaction was included in a finalized checkpoint:

    +-------------------------------------------------------------------+
    |                 TRANSACTION MERKLE TREE                           |
    +-------------------------------------------------------------------+
    |                                                                   |
    |                       tx_merkle_root                              |
    |                            /\                                     |
    |                           /  \                                    |
    |                    +-----+    +-----+                             |
    |                    | h01 |    | h23 |                             |
    |                    +--+--+    +--+--+                             |
    |                      /\          /\                               |
    |                     /  \        /  \                              |
    |                  +--+ +--+   +--+ +--+                            |
    |                  |h0| |h1|   |h2| |h3|                            |
    |                  +--+ +--+   +--+ +--+                            |
    |                   |    |      |    |                              |
    |                 [Tx0][Tx1]  [Tx2][Tx3]                            |
    |                                                                   |
    +-------------------------------------------------------------------+
    
    Proof for Tx2:
      - leaf_hash: h2 = SHA256(Tx2)
      - siblings:  [h3, h01]
      - path_bits: [left, right]
      - Compute: hash(h2, h3) -> h23
                 hash(h01, h23) -> root
      - Verify: root == checkpoint.tx_merkle_root

Transaction Inclusion Proof Structure:
    struct TxInclusionProof {
        tx_hash:          String
        checkpoint_height:u64
        merkle_proof:     [String]   // Sibling hashes bottom-up
        merkle_index:     u32        // Leaf position
    }


3.2.3 URL ENCODING FORMAT
--------------------------------------------------------------------------------

Self-contained proofs are compressed and encoded for URL embedding:

    Encoding Pipeline:
    
    +----------+     +---------+     +------------+     +-------------+
    | Proof    | --> | JSON    | --> | DEFLATE    | --> | Base64url   |
    | Struct   |     | Serialize|    | Compress   |     | Encode      |
    +----------+     +---------+     +------------+     +-------------+
                                                              |
                                                              v
                                                     /sp/{encoded_payload}

Base64url Alphabet:
    ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_

Typical proof sizes:

    +-------------------------------------------------------------------+
    |              PROOF SIZE REFERENCE                                  |
    +-------------------------------------------------------------------+
    | Proof Type       | Compressed  | Base64url    | Full URL          |
    |                  | (bytes)     | (chars)      | (w/ prefix)       |
    +------------------+-------------+--------------+-------------------+
    | Account proof    | 300-500     | 400-670      | 404-674 chars     |
    | Tx inclusion     | 200-400     | 267-534      | 271-538 chars     |
    | Weight proof     | 400-600     | 534-800      | 538-804 chars     |
    | Self-contained   | 800-1500    | 1067-2000    | 1071-2004 chars   |
    +------------------+-------------+--------------+-------------------+
    
    Compressed bytes = DEFLATE output
    Base64url chars  = ceil(bytes * 4/3)
    Full URL         = base64url chars + prefix (e.g., "/sp/" = 4 chars)
    
    All sizes fit within standard URL length limits (2048 chars) and
    QR code capacity (4296 alphanumeric chars at version 40).


3.3 ENCRYPTED MESSAGING (DOUBLE RATCHET PROTOCOL)
--------------------------------------------------------------------------------

Rinku's chat layer implements Signal-style Double Ratchet encryption for
end-to-end encrypted direct messages transported as DAG transactions.

    +-------------------------------------------------------------------+
    |              ENCRYPTED MESSAGE FLOW                                |
    +-------------------------------------------------------------------+
    
    Alice                          Validators                      Bob
      |                               |                             |
      | 1. ECDH key exchange          |                             |
      |   (P-256 ephemeral keys)      |                             |
      |<============================================>===============>|
      |                               |                             |
      | 2. Derive shared secret       |                             |
      |   (HKDF-SHA256)               |                             |
      |                               |                             |
      | 3. Encrypt message            |                             |
      |   (AES-256-GCM)              |                             |
      |                               |                             |
      | 4. Submit as data-only tx     |                             |
      |   (encrypted content in memo) |                             |
      |------>  tx{to: Bob,           |                             |
      |         memo: ciphertext}     |                             |
      |           |                   |                             |
      |           +-------> Gossip    |                             |
      |           |    (validators    |                             |
      |           |     transport     |                             |
      |           |     ciphertext    |                             |
      |           |     they CANNOT   |                             |
      |           |     decrypt)      |                             |
      |           |                   +---->  Bob decrypts          |
      |           |                   |       with ratchet key      |
      |                               |                             |
      | 5. Ratchet forward            |                             |
      |   (destroy used keys)         |                             |
      |   FORWARD SECRECY             |                             |
    
    +-------------------------------------------------------------------+

Privacy Properties:
    - CONTENT PRIVACY: AES-256-GCM encryption. Validators and other
      network participants see only ciphertext.
    - FORWARD SECRECY: Per-message key destruction via Double Ratchet.
      Compromising current keys cannot decrypt past messages.
    - KEY FRESHNESS: ECDH ratchet generates new key material with
      each message exchange, limiting the blast radius of any
      key compromise.
    - METADATA MINIMIZATION: Messages are standard data-only
      transactions. They look identical to any other zero-value
      transaction with a memo field.

On-Chain Message Format:
    A chat message is a standard Rinku transaction:
    - amount: 0 (data-only)
    - to: recipient address (or mailbox address)
    - memo: encrypted envelope containing:
        - ciphertext (AES-256-GCM encrypted message)
        - ratchet generation indicator
        - ephemeral public key (for ECDH ratchet step)
        - IV/nonce for decryption
    - kind: DataOnly

The compact envelope format ensures encrypted messages fit within
standard transaction size limits while carrying all cryptographic
material needed for decryption.

Remaining Metadata Leak: Timing Analysis

    Even with encrypted content and mailbox addresses, an observer who
    can see transaction timestamps can infer communication patterns:
    "someone sent a message at 14:03, and someone else sent one at 14:03:02"
    suggests a conversation.

    +-------------------------------------------------------------------+
    |              TIMING METADATA MITIGATIONS                          |
    +-------------------------------------------------------------------+
    
    Mitigation 1: BATCHING WINDOWS (low cost)
        Validators hold data-only transactions in a buffer and release
        them in fixed-interval batches (e.g., every 2-5 seconds).
        All messages in a batch appear simultaneous, destroying
        fine-grained timing correlation.
        Trade-off: adds 2-5s latency to message delivery.
    
    Mitigation 2: FIXED-RATE HEARTBEATS (medium cost)
        Clients send a steady stream of data-only transactions at a
        fixed rate (e.g., 1 per 10 seconds), whether or not they have
        real messages to send. Real messages replace heartbeats;
        dummy heartbeats are indistinguishable from real ones.
        Trade-off: increases transaction volume by ~6 tx/min/user.
    
    Mitigation 3: COVER TRAFFIC (high cost)
        The network generates synthetic transactions between random
        addresses to mask real traffic patterns. Effective but
        expensive in storage and gas.
        Trade-off: significant overhead, typically impractical.
    
    Recommended default: Batching windows (Mitigation 1) provide
    meaningful timing privacy at minimal cost and are compatible
    with fast-path finality (~2-5s batch vs ~10s checkpoint).
    
    +-------------------------------------------------------------------+


3.4 MAILBOX ADDRESSES (UPCOMING)
--------------------------------------------------------------------------------

Mailbox addresses are the missing piece for receiver privacy. Today,
encrypted messages protect content but expose the recipient's address
in the `to` field — creating a linkability problem.

    +-------------------------------------------------------------------+
    |              THE LINKABILITY PROBLEM                               |
    +-------------------------------------------------------------------+
    
    Without mailbox addresses:
    
    tx1: { from: Alice, to: 0xBob_Real, memo: [encrypted] }
    tx2: { from: Carol, to: 0xBob_Real, memo: [encrypted] }
    tx3: { from: Alice, to: 0xBob_Real, memo: [encrypted] }
    
    An observer can see:
    - Bob receives messages from Alice and Carol
    - Alice sent Bob two messages
    - Communication frequency and timing patterns
    
    +-------------------------------------------------------------------+
    
    With mailbox addresses:
    
    tx1: { from: Alice, to: 0xMailbox_A1, memo: [encrypted] }
    tx2: { from: Carol, to: 0xMailbox_C1, memo: [encrypted] }
    tx3: { from: Alice, to: 0xMailbox_A2, memo: [encrypted] }
    
    An observer sees:
    - Three unrelated transactions to three different addresses
    - No way to link them to the same recipient
    - No communication pattern visible
    
    +-------------------------------------------------------------------+

Planned Design:
    - Ephemeral receive addresses derived from a shared secret
      between sender and recipient
    - Each mailbox address is single-use or short-lived
    - Recipient can scan for messages using their private key
    - Mailbox addresses are unlinkable to the recipient's real address
      without the shared secret
    - Compatible with existing proof system (mailbox addresses
      participate in the state trie like regular accounts)

Message Discovery (How Does Bob Find His Messages?):

    The natural question: if Bob's messages go to ephemeral addresses
    that only he can derive, how does he find them?

    +-------------------------------------------------------------------+
    |              MAILBOX SCANNING MODES                                |
    +-------------------------------------------------------------------+
    
    SMALL-GROUP MODE (closed networks with <1000 participants):
    
        Bob derives the next N expected mailbox addresses from each
        active conversation's shared secret, then scans recent
        transactions for matches. In a bounded closed network, this
        is a small search space:
        
        Scan cost = (active conversations) × (lookahead window)
        Example:   20 conversations × 5 lookahead = 100 address checks
        
        This is acceptable because closed networks have bounded
        participant sets. Bob is scanning a small ledger, not
        all of Ethereum.
    
    LARGE-GROUP MODE (optional, for scaled deployments):
    
        Untrusted helper indexers maintain a mapping of mailbox
        addresses to encrypted notification hints. The indexer
        cannot read messages (content is encrypted) and cannot
        link mailbox addresses to real identities (only the
        recipient can derive the address). If the indexer lies
        about what's available, the recipient can verify against
        the ledger's Merkle proofs — the proofs keep indexers honest.
        
        Trust model: indexer can omit (censor) but cannot fabricate.
        Censorship is detectable by periodic full scans.
    
    +-------------------------------------------------------------------+

Combined with Double Ratchet encryption and closed-network boundaries,
mailbox addresses complete the privacy stack:
    - Content is encrypted (Double Ratchet)
    - Recipient is hidden (mailbox addresses)
    - Network boundary limits metadata leakage (closed networks)
    - Communication is verifiable (self-contained proofs)


3.5 CLOSED NETWORKS AS PRIVACY BOUNDARIES
--------------------------------------------------------------------------------

    +-------------------------------------------------------------------+
    |              CLOSED NETWORKS = PRIVACY GROUPS                     |
    +-------------------------------------------------------------------+
    |                                                                    |
    |  Rinku's closed-network model is not a limitation.                 |
    |  It is a privacy architecture decision.                            |
    |                                                                    |
    |  A closed network bounds the set of participants, which:           |
    |  - Limits metadata leakage (fewer addresses to correlate)          |
    |  - Makes traffic analysis harder (smaller, controlled set)         |
    |  - Enables practical partial-stateless verification                |
    |  - Creates natural privacy groups for communication                |
    |                                                                    |
    +-------------------------------------------------------------------+

For closed-network dApps where:
  - Wallets only interact with the dApp (no arbitrary external txs)
  - Known subset of addresses participate
  - No external addresses will interact unexpectedly
  - Validator set is known or changes infrequently

This enables MINIMAL state requirements:

    +-------------------------------------------------------------------+
    |           FULL NODE vs PARTIAL-STATELESS CLIENT                    |
    +-------------------------------------------------------------------+
    
    Full Node                          Partial-Stateless Client
    ---------                          ------------------------
    - Complete DAG history             - Only proofs for relevant accounts
    - All account states               - Only checkpoint headers
    - All transactions                 - Self-contained proofs for
    - All checkpoints                    transactions it cares about
    
    Storage: ~GB                       Storage: ~KB per user
    Sync: Minutes-Hours                Sync: Instant (proof-based)

Open-network stateless clients face an unsolvable bootstrapping
problem: verifying proofs requires knowing the current validator
set, which requires tracking global state, which defeats the
purpose of being stateless.

Closed networks sidestep this entirely. The validator set is known
at deployment time or changes through a governed process. This
makes the trust anchor static and the privacy boundary clear.


3.6 PARTIAL-STATELESS CLIENTS
--------------------------------------------------------------------------------

Within a closed network, clients need only the proofs relevant to
their interactions. This is privacy by architecture: you literally
do not have other participants' data.

Required Proofs by Use Case:

    +-------------------------------------------------------------------+
    |              PROOF REQUIREMENTS BY USE CASE                        |
    +-------------------------------------------------------------------+
    
    Use Case: Check my balance
    Required: AccountStateProof for my address
    
    Use Case: Verify payment received
    Required: TxInclusionProof + AccountStateProof (updated balance)
    
    Use Case: Check content trust score
    Required: TransactionWeightProof
    
    Use Case: Full offline verification
    Required: SelfContainedProof (includes BLS sig + validator set)

Verification flow for partial-stateless client:

    +-------------------------------------------------------------------+
    |           OFFLINE VERIFICATION FLOW                                |
    +-------------------------------------------------------------------+
    
    Step 1: Receive proof (via URL, API, or direct)
            |
            v
    Step 2: Parse & validate structure
            |
            +-----> Check version, required fields
            |
            v
    Step 3: Verify Merkle inclusion
            |
            +-----> Reconstruct leaf hash from data
            +-----> Walk proof path using siblings
            +-----> Compare computed root vs checkpoint root
            |
            v
    Step 4: Verify checkpoint authority (optional but recommended)
            |
            +-----> Parse BLS aggregated signature
            +-----> Parse signer bitmap
            +-----> Verify aggregate sig against checkpoint hash
            +-----> Check signer stake >= 2/3 threshold
            |
            v
    Step 5: PROOF VERIFIED - trust the data
    
    
    No network calls required. No full node required.
    Only need: proof + known chain identity (genesis hash)

Trust Anchors for Closed Networks:

    +-------------------------------------------------------------------+
    |              TRUST ANCHOR TABLE                                    |
    +-------------------------------------------------------------------+
    | Anchor               | What it proves         | Source             |
    +----------------------+------------------------+--------------------+
    | Genesis checkpoint   | Chain identity          | Hardcoded in dApp  |
    | hash                 | (am I on the right      | at deployment      |
    |                      |  ledger?)               |                    |
    +----------------------+------------------------+--------------------+
    | Initial validator    | Signer authority        | Hardcoded or       |
    | public keys (BLS)    | (who is allowed to      | governed update    |
    |                      |  sign checkpoints?)     | mechanism          |
    +----------------------+------------------------+--------------------+
    | Latest checkpoint    | Freshness               | Received from      |
    | height (or head cert)| (is this proof current  | any peer or API;   |
    |                      |  or replayed?)          | verifiable via sig |
    +----------------------+------------------------+--------------------+
    
    In a closed network, these three anchors are static or slow-changing.
    The client embeds them at build time and updates them only through
    a governed process (e.g., validator rotation proposal). This is why
    partial-stateless verification works: the trust root doesn't move.


================================================================================
               IV. TRUST: ACCOUNTABILITY WITHOUT IDENTITY
================================================================================

Privacy without accountability enables abuse. Rinku's weight attestation
system provides a mechanism for community trust scoring that works without
requiring participants to reveal their identity — only their stake.

A staker can vote on content quality (boost, suppress, neutral) and their
vote is weighted by stake, not identity. The result is a reputation system
for content, not for people.


4.1 WEIGHT ATTESTATION SYSTEM
--------------------------------------------------------------------------------

Stakers can vote on any transaction's "weight" — a protocol-level trust score.
This is designed for applications like content moderation, anti-disinformation,
and community curation where accountability matters but identity should not
be required.

    Privacy-Preserving Properties:
    - Votes are linked to stake, not to human identity
    - Aggregated results reveal community consensus, not individual votes
    - Weight proofs can be verified offline (same as other proofs)
    - No central moderator — trust emerges from stake-weighted consensus


4.2 VOTE TYPES & STAKE-WEIGHTED AGGREGATION
--------------------------------------------------------------------------------

    enum WeightVote {
        Boost,      // Endorses transaction/content (+weight)
        Suppress,   // Flags transaction/content (-weight)
        Neutral     // Abstains from judgment (0 weight)
    }

Vote Structure:
    struct PendingWeightVote {
        tx_hash:          String    // Transaction being voted on
        validator_pubkey: String    // Voter's public key (130 hex chars)
        vote:             WeightVote
        timestamp:        u64
        signature:        [u8]      // BLS signature of vote
    }

Votes are aggregated by stake weight at checkpoint finalization:

    +-------------------------------------------------------------------+
    |              VOTE AGGREGATION EXAMPLE                              |
    +-------------------------------------------------------------------+
    
    Transaction: 0xabc123...
    
    Voters:
    +----------------+--------+--------+
    | Address        | Stake  | Vote   |
    +----------------+--------+--------+
    | 0x1234...      | 1000   | Boost  |
    | 0x5678...      |  500   | Boost  |
    | 0x9abc...      |  200   | Suppress|
    | 0xdef0...      | 1000   | Neutral|
    +----------------+--------+--------+
    
    Aggregation Result (1 RKU = 100,000,000 micro):
    +----------------------+----------------+
    | boost_stake_micro    | 150,000,000,000| (1500 RKU)
    | suppress_stake_micro |  20,000,000,000| (200 RKU)
    | neutral_stake_micro  | 100,000,000,000| (1000 RKU)
    | net_weight           | 130,000,000,000| (boost - suppress)
    | attestation_count    |              4 |
    | total_network_stake  | 270,000,000,000|
    +----------------------+----------------+

Trust Score Calculation:
    net_ratio = net_weight / total_network_stake
    Example: 130B / 270B = 0.48 (48% net positive)


4.3 WEIGHT TRIE STRUCTURE
--------------------------------------------------------------------------------

Aggregated weights are stored in a Merkle trie for offline verification:

                         WEIGHT TRIE ROOT
                              |
                    +---------+---------+
                    |                   |
               [subtree]           [subtree]
                    |                   |
              +-----+-----+       +-----+-----+
              |           |       |           |
           [leaf]      [leaf]   [leaf]      [leaf]
              |           |       |           |
           Tx A        Tx B    Tx C        Tx D
           weights     weights weights     weights

Weight Trie Leaf (for each voted transaction):
    struct WeightTrieLeaf {
        tx_hash:                  String
        boost_stake_micro:        u64
        suppress_stake_micro:     u64
        neutral_stake_micro:      u64
        total_network_stake_micro:u64
        attestation_count:        u32
    }

Leaf Hash:
    SHA256("weight:{tx}:{boost}:{suppress}:{neutral}:{total}:{count}")

The weight_trie_root is included in each checkpoint, enabling:
  - Offline verification of any transaction's trust score
  - Compact proofs for content moderation decisions
  - Historical trust score auditing


4.4 WEIGHT STATE PROOFS (TransactionWeightProof)
--------------------------------------------------------------------------------

For closed-network dApps with trust scoring, weight proofs verify the
community's assessment of a transaction.

    struct TransactionWeightProof {
        tx_hash:                  String
        checkpoint_height:        u64
        
        // Aggregated weight data
        boost_stake_micro:        u64    // Stake that voted "boost"
        suppress_stake_micro:     u64    // Stake that voted "suppress"
        neutral_stake_micro:      u64    // Stake that voted "neutral"
        net_weight:               i64    // boost - suppress
        attestation_count:        u32    // Number of unique voters
        total_network_stake_micro:u64    // Denominator for ratios
        
        // Weight Trie inclusion proof
        weight_trie_root:         String
        merkle_proof:             [String]
        leaf_index:               u32
    }

Weight Trie Leaf Hash:
    SHA256("weight:{txHash}:{boost}:{suppress}:{neutral}:{total}:{count}")

5.4 GAS FEE MODEL
--------------------------------------------------------------------------------

Dynamic EIP-1559 style gas pricing with validator/burn split:

    Gas Price Configuration:
    +------------------------+------------------+
    | Minimum gas price      | 0.001 RKU        |
    | Maximum gas price      | 10.0 RKU         |
    | Target txs per period  | 3000 txs         |
    | Period duration        | 15 seconds       |
    | Adjustment factor      | 12.5% max/period |
    +------------------------+------------------+
    
    Effective target: ~200 TPS (3000 txs / 15s period)

Fee Processing (two-layer system, applied sequentially):

    Layer 1 - GasService (per-transaction burn, applied first):
        burn_amount = fee * min((1 - min_gas_price) * supply_ratio, 0.3)
        This burns a portion of each gas fee at transaction time.
        Remaining fee (fee - burn_amount) enters the reward pool.
    
    Layer 2 - EmissionService (checkpoint reward distribution):
        Distributes accumulated fee pool + emission rewards at each checkpoint.
        Validators receive minimum 70% (VALIDATOR_FEE_FLOOR_PERCENT)
        Additional burn receives up to 30% (BURN_CEILING_PERCENT)
        Split adjusts based on circulating supply ratio
        At 50% circulation (SUPPLY_TARGET_FOR_FULL_BURN), burn reaches ceiling
    
    +-------------------------------------------------------------------+
    |                FEE SPLIT CURVE                                     |
    +-------------------------------------------------------------------+
    |                                                                    |
    | Burn %  |                                          ____           |
    |   30%   |                                     ____/               |
    |   20%   |                                ____/                    |
    |   10%   |                           ____/                         |
    |    0%   |________________________/                                |
    |         +-------------------------------------------->            |
    |         0%           25%           50%           75%   Circ/Max   |
    |                                                                   |
    +-------------------------------------------------------------------+

Gas Price Adjustment (from GasService):
    - Tracks transaction counts per period (15s windows)
    - Adjusts price by up to +/-12.5% based on demand vs target
    - Stays within min (0.001 RKU) and max (10.0 RKU) bounds

Burn Amount Calculation (per transaction):
    burn_ratio = (1.0 - min_gas_price) * supply_ratio
    burn_ratio = min(burn_ratio, 0.3)   // Cap at 30%
    burn_amount = fee * burn_ratio
    
    Where supply_ratio = circulating_supply / max_supply


================================================================================
       VI. PROOF-CARRYING RECEIPTS AS A LEDGER-INDEPENDENT PATTERN
================================================================================

6.1 THE PATTERN
--------------------------------------------------------------------------------

Rinku's self-contained proof system implements a general pattern that we call
"proof-carrying receipts." The core idea: a receipt artifact carries everything
needed to verify a fact about the ledger, without contacting the ledger.

    +-------------------------------------------------------------------+
    |              PROOF-CARRYING RECEIPT PATTERN                        |
    +-------------------------------------------------------------------+
    |                                                                    |
    |                      +------------------+                          |
    |                      | PROOF-CARRYING   |                          |
    |                      | RECEIPT          |                          |
    |                      +------------------+                          |
    |                              |                                     |
    |         +--------------------+--------------------+                |
    |         |                    |                    |                 |
    |         v                    v                    v                 |
    |  +--------------+   +--------------+   +------------------+        |
    |  | Finality     |   | State/Tx     |   | Signer Set       |       |
    |  | Certificate  |   | Commitment   |   | Membership Proof |       |
    |  | (quorum sig) |   | Root         |   | (optional*)      |       |
    |  +--------------+   +--------------+   +------------------+        |
    |         |                    |                    |                 |
    |         +--------------------+--------------------+                |
    |                              |                                     |
    |                    +---------+---------+                            |
    |                    | Canonical Encoding|                            |
    |                    | (deterministic)   |                            |
    |                    +-------------------+                            |
    |                              |                                     |
    |                              v                                     |
    |                   PORTABLE PROOF ARTIFACT                          |
    |                   (URL, QR code, message)                          |
    |                                                                    |
    +-------------------------------------------------------------------+
    
    * Required for full offline verification (Profile C).
      Without it, the verifier must already know the signer set.

    Primary system:     DAG + checkpoints (standard distributed ledger)
    Redundancy layer:   Proof-carrying receipts that can be transported
                        via URL, QR code, or message and verified offline


6.2 REQUIRED PRIMITIVES
--------------------------------------------------------------------------------

Any ledger implementing proof-carrying receipts needs these four primitives:

    +-------------------------------------------------------------------+
    |              REQUIRED PRIMITIVES                                   |
    +-------------------------------------------------------------------+
    
    (i) FINALITY CERTIFICATE
        A portable quorum signature that a verifier can check offline.
        Rinku uses: BLS12-381 aggregated signatures + signer bitmap +
        2/3 stake threshold check.
        
        Why non-trivial: Many networks don't have a clean "portable
        quorum cert" format that's stable and easy to embed. They
        assume the client will query the network for finality status.
    
    (ii) STATE/TX COMMITMENT ROOT
        A cryptographic commitment (Merkle root) to the state or
        transaction set at finality time.
        Rinku uses: Sparse Merkle Trie for state, standard Merkle
        tree for transactions, MerkleSumTree for validator weights.
    
    (iii) SIGNER SET MEMBERSHIP PROOF
        A way to prove the signer set and their weights WITHOUT
        querying the chain. This is the hardest piece.
        Rinku uses: MerkleSumTree multi-proofs that include validator
        leaves, their BLS public keys, and stake weights, all provable
        against the checkpoint's validator_sum_tree_root.
        
        Why non-trivial: Most systems assume the client can fetch
        validator sets (or sync committees) from the network. Embedding
        the proof of WHO signed and THEIR authority is what makes
        full offline verification possible.
    
    (iv) CANONICAL DETERMINISTIC ENCODING
        A stable encoding so proofs are cross-language verifiable.
        Rinku uses: Decimal strings for amounts/weights (avoiding
        float drift), JSON serialization, DEFLATE compression,
        base64url encoding.
        
        Why non-trivial: Many "receipt" ideas die at this step.
        IEEE 754 floats serialize differently across languages,
        JSON key ordering varies, and compression algorithms may
        produce non-deterministic output. All of these break
        signature verification.


6.3 WHAT URLs SOLVE (AND DON'T)
--------------------------------------------------------------------------------

    +-------------------------------------------------------------------+
    |              URLs: VERIFICATION, NOT DISCOVERY                     |
    +-------------------------------------------------------------------+
    |                                                                    |
    |  URLs SOLVE:                                                       |
    |  - Verification AFTER you have the artifact                        |
    |  - Portable transport (paste, share, embed, QR)                    |
    |  - Self-describing format (URL path indicates proof type)          |
    |  - Browser-native handling (no special software)                   |
    |                                                                    |
    |  URLs DO NOT SOLVE:                                                |
    |  - Discovery of new transactions                                   |
    |  - Real-time state synchronization                                 |
    |  - Peer-to-peer gossip                                             |
    |  - Consensus participation                                         |
    |                                                                    |
    +-------------------------------------------------------------------+

This is exactly how a redundancy feature should behave: the primary system
(DAG + checkpoint consensus) handles discovery, ordering, and finality. The
proof-carrying receipt layer provides an independent verification path that
works even when the primary system is unreachable.


6.4 ADOPTION DIFFICULTY FOR EXISTING DAGs
--------------------------------------------------------------------------------

Many DAGs already have the raw ingredients for proof-carrying receipts:
  - Transaction hashes and ancestry
  - Periodic finality anchors (or could add them)
  - Some form of authenticated validator attestations

But adopting the pattern is NOT a trivial side-effect. It typically requires
explicit engineering in three areas:

    +-------------------------------------------------------------------+
    |              ADOPTION DIFFICULTY MATRIX                            |
    +-------------------------------------------------------------------+
    
    Primitive                     | Difficulty | Why
    ------------------------------|------------|----------------------------
    Portable quorum cert format   | HIGH       | Most networks assume online
                                  |            | finality queries; packaging
                                  |            | a stable, embeddable cert
                                  |            | requires new message types
    ------------------------------|------------|----------------------------
    Signer set membership proof   | HIGH       | Proving WHO signed and
                                  |            | their authority without
                                  |            | chain access requires
                                  |            | MerkleSumTree or similar
                                  |            | commitment structures
    ------------------------------|------------|----------------------------
    Canonical encoding            | MEDIUM     | Cross-language determinism
                                  |            | for all proof fields;
                                  |            | decimal strings, sorted
                                  |            | keys, stable compression
    ------------------------------|------------|----------------------------
    State/Tx commitment roots     | LOW        | Most DAGs already have
                                  |            | Merkle roots or can add
                                  |            | them easily
    ------------------------------|------------|----------------------------

Rinku implements all four primitives natively within a DAG + checkpoint
design, making proof-carrying receipts a first-class feature rather than
a retrofit.
