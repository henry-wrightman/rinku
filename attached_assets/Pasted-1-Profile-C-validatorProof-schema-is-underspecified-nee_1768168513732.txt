1) Profile C validatorProof schema is underspecified (needs enough to actually reconstruct)

Right now you have:

validatorProof: {
  signerLeaves: Array<{ pubKey: string; weight: uint64 }>;
  auxiliaryNodes: string[];
  totalWeight: uint64;
}


…but for MerkleSumTree multiproofs, you generally need:

the indices of signer leaves (or a rule that derives them from the bitmap), and

enough structure to reconstruct internal nodes deterministically:

either (level,index) per auxiliary node, or

a canonical “flags” multiproof format, or

a deterministic sparse-map rule that implies positions.

You mention “signerLeaves ordered by increasing index from signerBitmap popcount walk” — great — but then you still need to specify how auxiliaryNodes map to positions. “string[]” alone is ambiguous.

✅ Fix: define auxiliaryNodes as either:

Array<{ level: uint8; index: uint16; hash: string; sumWeight: uint64 }> (explicit), or

a packed multiproof encoding spec (flags + nodes), but then say that in the schema (even briefly).

Without this, verifyValidatorProof() cannot be implemented from the paper.

2) Profile B “Merkle-included” claim needs the Merkle path (or clarify it’s deferred)

Profile B currently includes txMerkleRoot but no inclusion path in the bundle. So strictly speaking Profile B can’t prove inclusion from the URL alone unless you assume:

inclusion is implied by ancestry (not true), or

inclusion proof is fetched externally (breaks the “URL carries proof” story), or

the Merkle path is embedded somewhere else (not shown).

✅ Fix options:

Add txMerklePath: string[] + txMerkleIndex: uint16 to Profile B checkpoint object; or

Reword Profile B as “anchored to a checkpoint header commitment; inclusion proof is optional / may be included.”

Given your thesis, I’d include the path.

4) Canonical JSON + hash definition should mention “exclude hash field”

You do hash = sha256(canonical tx JSON), but the ProofBundle includes the hash itself. Make sure you explicitly say hash is computed over the tx object only (not including bundle fields).

(Your code does this already, just clarify in text.)

Your Profile C verifier checks:

but in the pseudocode you don’t explicitly sum weights vs totalWeight. (Maybe inside verifyValidatorProof.)
