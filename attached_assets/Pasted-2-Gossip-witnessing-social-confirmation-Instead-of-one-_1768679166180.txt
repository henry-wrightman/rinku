2) Gossip “witnessing” / social confirmation

Instead of one authoritative certificate, you accept something as real because you saw it propagated widely, or because multiple independent peers report it.

Examples:

P2P networks: “I saw this message from many peers”

Some DAG consensus designs: acceptance confidence grows as more later messages reference it

Trust model: assumes you’re not eclipsed/isolated, and that the network is sufficiently connected.

Downside: hard to turn into a portable, offline proof. Great for “feel confident,” weaker for “prove to a third party later.”

3) Content-addressed logs / transparency proofs

You don’t prove “this is final in a global ledger,” you prove:

“This event was committed to an append-only log with root R, signed by the log.”

This is the Certificate Transparency / Merkle tree log pattern:

you get an inclusion proof + a signed tree head (STH)

Trust model: trust the log operator(s), or use multiple logs + auditing.

Downside: it’s not a decentralized consensus ledger by itself unless you decentralize the log (or anchor it elsewhere).

4) Time-stamping / anchoring into a stronger system

You prove “this existed by time T” by anchoring it into something widely trusted:

Bitcoin timestamping, etc.

Trust model: trust the anchoring chain’s security; verification usually requires chain headers (or a trusted checkpoint).

Downside: proves existence/time, not necessarily “valid state transition.”

5) Payment-channel / federated receipts

In channels, the “valid” state is the latest signed state between parties. You don’t need the chain unless there’s a dispute.

Trust model: trust the counterparty signature + your own stored latest state.

Downside: doesn’t generalize to global validity unless you have dispute resolution.

6) Trusted execution / hardware attestation

You accept a result as genuine because it came from a trusted hardware enclave with remote attestation.

Trust model: trust the hardware vendor + attestation infrastructure.

Downside: not censorship-resistant; introduces vendor trust.

7) ZK proofs of validity

You accept a transaction/state transition because there’s a proof that the transition is valid under some rules, and the proof is anchored to a known commitment/root.

This is how validity rollups work, and it’s also how “proof-carrying computation” works.

Trust model: trust the proving system + verification key + anchoring commitment.

Downside: proofs usually don’t prove “final in a decentralized network” unless combined with a finality anchor.
