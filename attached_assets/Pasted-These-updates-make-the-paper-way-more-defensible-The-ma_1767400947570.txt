Suggestion (tight wording fix):
Change the claim to something like:

“No trusted infrastructure required for verification; availability may be provided by the sender or the network. Proof bundles can be self-contained up to the chosen proof profile.”

And define “Proof Profile A (QR)” vs “Proof Profile B (Full finality cert)”:

QR-profile: includes checkpoint id + signatureCount only (fast UX, weaker)

Full-profile: includes aggregated signature / bitmap / validator-set proof (strong, larger)

Right now you’re mixing “full proof” language with a deliberately trimmed payload.

So, as written, the verifier can validate:

ancestry consistency

hash integrity

…but not “tx included in checkpoint” unless you either:

include a Merkle path (txHash → txMerkleRoot), and

include a verifiable checkpoint certificate (signatures/aggregate sig + membership proof), or

treat checkpoints as “known trusted objects” fetched externally.

Suggestion: Make inclusion proofs optional and explicit:

add txInclusionProof: { path: string[], index: number } (or compact bit/path encoding)

add checkpointCert: { aggSig, signerBitmap, validatorSetRoot } (if you want real finality-in-the-link)

If you don’t want that bloat, then call it what it is: “receipt that chains to a checkpoint anchor,” not “cryptographic finality proof.”
