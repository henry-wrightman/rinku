The main fixes I’d do (small, but important)
1) Your Profile definitions are slightly inconsistent

On page 4 you list:

Profile A: rinku://tx/{payload}

Profile B: rinku://txp/{payload}

Profile C: rinku://sp/{payload} 

Md2File.com document.pdf (1)

But earlier you described txp as “complete proof bundle,” while Profile A is also called receipt. Right now it’s a little unclear what the difference is between tx vs txp (especially since the proof bundle schema includes parents + checkpoint). 

Md2File.com document.pdf (1)

Fix: Make the profiles purely about what’s included, not the URL path name:

tx = just tx + sender pubkey + tx sig + tx hash (no parents)

txp = tx + parents recursion + checkpoint anchor (your “receipt with ancestry”)

sp = txp + full finality cert components (BLS agg sig + signer bitmap + validator proof)

2) The appendix schema still uses number for amounts/fees

Your main paper talks about smallest units, but Appendix A.2 says amount: number; fee: number; ... 

Md2File.com document.pdf (1)

That’s a spec footgun if you want this to be adoptable. JS numbers are floats; other languages will interpret them differently.

Fix: change to:

amount: uint64

fee: uint64

nonce: uint64

ts: uint64
and state “encoded as decimal strings in JSON canonical form” if needed.

3) “Self-contained proofs” still assume a trust anchor

You do mention bootstrapping in §6 (good). 

Md2File.com document.pdf (1)


But you also say “Profile C proves finality without any external trust anchors beyond knowing which network you are verifying against.” 

Md2File.com document.pdf (1)

Even in your own trust bootstrapping section, a fresh verifier needs genesis keys or a pinned checkpoint. 

Md2File.com document.pdf (1)

Fix: one line:
“Profile C requires a pinned chain identity + initial trust anchor (genesis/pinned checkpoint); after bootstrapping, proofs verify offline.”

4) Add verifier safety limits (DoS protection)

Since your verifier recursively verifies parents, you should specify:

max depth

max total decoded bytes

max parents per tx (you say 0–2 tips, but parent bundles are still recursive) 

Md2File.com document.pdf (1)

Put this as a bullet list in “Limitations” or “Verification Algorithm.”

This is a huge “adoptability” detail because anyone embedding this in a browser will need it.

5) Minor terminology: “PoS receipts” typo

You write “PoS receipts” under Profile A use case (page 4). Probably meant POS.
