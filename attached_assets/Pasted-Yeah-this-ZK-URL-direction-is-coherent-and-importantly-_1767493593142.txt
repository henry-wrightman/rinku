The main things I’d tighten (these matter)
1) QR “alphanumeric mode” claim: beware the rinku://zkq/ prefix

Even if the payload is base45, the literal string rinku://zkq/ contains lowercase letters and ://, which often forces byte-mode segments depending on the QR encoder (some encoders don’t optimally segment modes). So “Base45 stays in alphanumeric mode” can be true in theory, but false in practice if libraries don’t segment well. 

WHITEPAPER (6)

Fix options (pick one):

QR encodes only the base45 blob (no prefix), and your scanner/app prepends rinku://zkq/ internally.

Use a super-short prefix like rk: (still byte-mode likely, but fewer bytes).

In the paper, phrase it as: “base45 enables alphanumeric-mode payload density; actual capacity depends on encoder segmentation.”

2) Receipt theft / “bearer receipt” risk (still exists)

Recipient-binding stops cross-recipient reuse, but doesn’t stop someone else from presenting the same receipt to the same recipient (e.g., stolen QR/screenshot) before the buyer does.

You already have nullifiers, but you note offline cache is local-context only.
