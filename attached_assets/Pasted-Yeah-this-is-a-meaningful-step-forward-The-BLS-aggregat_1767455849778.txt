The main blocker: totalWeight can be lied about in Profile C

Right now an attacker can create a “valid” Profile C where:

they include only their own signer witnesses (real keys, real sig)

set totalWeight to something tiny

then “signerWeight ≥ 67% of totalWeight” passes easily

Your design prevents forging signer weights (because signer witnesses are bound via validatorSetRoot in the signed hash), but it does not prevent forging the denominator (totalWeight) if the verifier has no independent source for totalWeight.

So: Profile C is “offline verifiable quorum of included witnesses,” not “offline verifiable quorum of the actual validator set.”

You already say this, but if you want Profile C to be the flagship “no light node” story, you should close this gap.

How to fix Profile C without exploding size
Option 1 (best): Commit to the full validator set root and include a membership proof for signers

Instead of trusting totalWeight, make totalWeight derivable from the full validator set commitment.

Mechanism:

Each checkpoint commits to a validatorSetRoot of all validators (or a “validatorSetCommitment”).

In the proof, for each signer you include:

signer witness (index, pubkey, weight)

and a Merkle proof that this signer witness is included in the validatorSetRoot

Then the verifier knows signers are real members of the set.

But you still need the denominator: total stake weight. Two ways:

include totalWeight in the signed hash and

include an additional proof that totalWeight equals “sum of weights in validatorSetRoot.”
That’s hard to prove succinctly with a basic Merkle tree.

So Option 1 is good for “membership,” but not for “sum correctness” unless you change the commitment.

Option 2 (pragmatic + compact): Use a commitment that makes totalWeight verifiable

You can structure the validator set commitment so the verifier can compute/verify totalWeight without listing everyone.

Examples:

Store a tree where internal nodes include both hash and sumWeight, and the root commits to both. Then any membership proof can also carry partial sums, and the verifier can trust the root’s sumWeight if the root is signed.
This is essentially a “Merkle sum tree.”

Then Profile C can include:

signed root (already)

rootSumWeight

and for each signer: membership proof that also exposes signer weight (or binds it)

Now the verifier doesn’t have to “trust” totalWeight; it is part of the signed commitment and consistent with the sum-tree structure. That closes the denominator attack cleanly.

This is the single most important change if your goal is “self-provable offline” without a pinned validator set.

Option 3: Just embed the full validator set for that checkpoint

If your validator set is small (say 21–64) and you really want pure simplicity, you can include all validator witnesses in the proof. That makes totalWeight trivially checkable. But it likely pushes you out of QR territory quickly because each witness includes a 96-byte pubkey + address + weight.
